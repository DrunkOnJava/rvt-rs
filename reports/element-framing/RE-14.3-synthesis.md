# RE-14.3 synthesis — ArcWall record wire format fully decoded

**Date:** 2026-04-21
**Scope:** Hex-dump 32 filtered ArcWall (tag 0x0191) records on Einhoven Partitions/5. Identify record envelope, header structure, coordinate encoding, and record variants.

## TL;DR

**ArcWall records are directly decodable.** Standard wall records have a
16 B fixed header + 6 f64 coordinates + 6 f64 geometry duplicate + trailer.
Records are 292 B (single) or 568 B (double). 32 records on Einhoven
Partitions/5 all decode cleanly. The record variant byte (`0x07fa` for
standard, `0x0821` for compound) cleanly discriminates wall types.

This is the first **fully-decoded element record in rvt-rs**. Unblocks:
- `elements::decoders::arc_wall` (DEC-01)
- `walker::iter_partition_elements()` with actual decoded elements (RE-15)
- IFC exporter emitting >0 `IFCWALL` entities on real corpus (DEC-05)

## Record envelope — standard variant (records #4-#11, #14-#20)

```
Offset  Size  Field                      Observed value
+0x00   2     u16 tag                    0x0191 (ArcWall)
+0x02   2     u16 filter/pad             0x0000 (record-prefix filter)
+0x04   4     u32 fixed_header_0         0x00088004 ("class marker A")
+0x08   4     u32 fixed_header_1         0x00000001 ("count/version 1")
+0x0c   4     u32 fixed_header_2         0x00000003 ("type 3 = wall?")
+0x10   2     u16 variant_marker         0x07fa (standard), 0x0821 (compound)
+0x12   8     f64 coord_1                x-component
+0x1a   8     f64 coord_2                y-component
+0x22   8     f64 coord_3                z-component or offset
+0x2a   8     f64 coord_4                second x-component
+0x32   8     f64 coord_5                second y-component
+0x3a   8     f64 coord_6                second z-component or offset
+0x42   48    f64 × 6                    DUPLICATE of coords 1-6
+0x72   1     u8 trailer_0x03            0x03 (record terminator)
+0x73  varies padding or next record
```

Total core record: 115 B fixed + variable trailer. Records pack with 292 B
centre-to-centre spacing (for singletons) or 568 B (for pairs), giving
either 177 B or 453 B of additional content per record after the core.

## Record variant — compound (records #12, #13)

```
Offset  Size  Field                      Observed
+0x00   2     u16 tag                    0x0191
+0x02   2     u16 filter/pad             0x0000
+0x04   4     u32 fixed_header_0         0x00088004
+0x08   4     u32 fixed_header_1         0x00000003 ("version/count 3")
+0x0c   4     u32 fixed_header_2         0x00000003
+0x10   2     u16 variant_marker         0x0821 (compound)
+0x12   ...   variable-length sub-records delimited by 0x0821 0xNNNN / 0x0870 markers
```

Record #12 body contains three `21 08 XX 00 00 00` markers and a `70 08`
sub-marker. This is a compound wall (wall with door/window openings
or a wall assembly). Full decode deferred — the 30 singleton walls
cover the majority case.

## Record variant — index (record #0)

```
+0x00: 91 01 00 00
+0x04..: cf 02 00 00  d0 02 00 00  d2 02 00 00  e2 02 00 00 ...
```

Pure list of u32 values (0x02cf, 0x02d0, 0x02d2, 0x02e2, ..., 0x061e, 0x061f, ...,
0x0635). These look like IDs referencing other records or ElementIds. Possibly
a manifest of "what wall instances exist" used as a pre-index before the
actual wall data starts at record #4.

## Record variant — metadata (record #1)

```
+0x00: 91 01 00 00 00 00 00 00 ...(mostly zeros)...
+0x1d..: 95 00 00 00 00 bb 20 00 00 63 00 00 00 76 05 00 00 ...ff ff...
+0x4b..: 82 05 03 00 00 00 a5 20 00 00 ba 20 00 00 bb 20 00 00
+0x66..: 01 00 00 00 94 1d 00 00
```

Contains the **0x00000576 (1398)** and **0x00001d94 (7572)** constants we
discovered in RE-14.1's HostObjAttr analysis. **This is cross-corpus
coherence**: the same structural constants appear in both HostObjAttr
and ArcWall record variants, confirming these values are global
file-level identifiers (likely schema-version or class-family stamps)
that appear wherever records reference the ArcWall family.

This is evidence that **0x576 and 0x1d94 are class-family / schema-family
identifiers, not per-record data.** They will show up in multiple record
types across the schema as structural invariants.

## Record variant — minimal (records #2-#3)

Similar to #1 but with additional u32 references at specific positions.
Likely header records for a wall "group" or "worksharing batch."

## Record-count coherence

32 filtered ArcWall records on Einhoven Partitions/5:
- 1 index (#0) — manifest
- 3 metadata (#1, #2, #3) — "group headers"
- 26 standard walls (#4-#11, #14-#20, and similar-pattern records after)
- 2 compound walls (#12, #13)

Which leaves **~28 architectural wall elements** decodable. This is
within the same order of magnitude as the ~1411 ElemTable total
element count — Einhoven is an IFC5 sample model, likely with ~20-50
walls total, matching this figure.

## Rust decoder sketch

```rust
/// Decoded ArcWall element from Partitions/N wire format.
/// See `reports/element-framing/RE-14.3-synthesis.md` for the byte-level
/// wire format this maps onto.
pub struct ArcWallRecord {
    pub tag: u16,           // 0x0191
    pub fixed_header_0: u32,   // 0x00088004 (class-family marker)
    pub count_version: u32,    // 1 for standard, 3 for compound
    pub type_code: u32,        // 0x00000003 (wall type?)
    pub variant: u16,          // 0x07fa standard, 0x0821 compound
    pub coords: [f64; 6],      // primary geometry
    pub coords_dup: [f64; 6],  // duplicate (we assert equality)
    pub trailer: u8,           // 0x03
}

impl ArcWallRecord {
    /// Decode a standard ArcWall record at `buf[offset..]`.
    /// Returns Some only when the tag and filter pad match AND
    /// the variant marker is 0x07fa. Caller is responsible for
    /// scanning Partitions/* and positioning `offset` at real
    /// (record-prefix-filtered) occurrences.
    pub fn decode_standard(buf: &[u8], offset: usize) -> Option<Self> {
        if offset + 0x7a > buf.len() { return None; }
        let tag = u16::from_le_bytes([buf[offset], buf[offset + 1]]);
        if tag != 0x0191 { return None; }
        let filter = u16::from_le_bytes([buf[offset + 2], buf[offset + 3]]);
        if filter != 0 { return None; }
        let variant = u16::from_le_bytes([buf[offset + 0x10], buf[offset + 0x11]]);
        if variant != 0x07fa { return None; }
        // … read 6 f64 at +0x12..0x42, 6 more at +0x42..0x72 …
        // … assert duplicate match …
        Some(/* constructed record */)
    }
}
```

## Hypotheses

**H14 (new, conf 0.95)** — Element records in partitions are **self-describing
by tag + variant** with fixed-size bodies. The variant byte at +0x10 relative
to the tag determines the record layout. For ArcWall tag 0x0191:
- variant 0x07fa = standard wall (2 line segments, 6 coords × 2)
- variant 0x0821 = compound wall (embedded openings)
Other tags will have their own variant byte conventions.

**H15 (new, conf 0.9)** — The 0x00088004, 0x576, and 0x1d94 constants are
**schema-family identifiers** stable across record types within a family.
If we look for them in the Formats/Latest decoded schema, they should appear
as class IDs, ancestor IDs, or field-id markers for the HostObj / ArcWall
family of classes.

**H16 (new, conf 0.75)** — Coords arrive in pairs (3 + 3) that represent
**two points in 3D space** (start / end of a line segment that is the wall's
centerline). The "duplicate" block is not a true duplicate — it's the same
geometry expressed in a different reference frame (e.g., location-line vs
base-line). On walls where Z=0 and the duplicate matches the original,
it's because the wall is 2D and no re-projection is needed.

## Decisions

**D19** — Build `elements::decoders::arc_wall` as the first concrete
element decoder. Ship with tests against the 28 standard records on
Einhoven Partitions/5. Estimate: the decoder is ~40 LOC + tests.

**D20** — Build `walker::iter_arc_walls()` as a stand-in for the generic
`iter_partition_elements()` from RE-15. Walk Partitions/5, find records
with tag 0x0191 + variant 0x07fa, yield `ArcWallRecord`s.

**D21** — Wire ArcWallRecord → IFCWALL in the IFC exporter. The decoded
coords give us endpoint geometry directly; emit as `IFCBUILDINGELEMENTPROXY`
with name "ArcWall" (without full Wall type resolution) OR as `IFCWALL`
(fuller semantic).

**D22** — DEC-05 (IfcWall count > 0 on real file) is now unblocked.
28 standard walls on Einhoven → 28 IFCWALL entities emitted → test
can assert count ≥ 1.

## Open questions

**Q15** — What do the coord semantics actually mean? Are they:
  - `(start_xyz, end_xyz)` centerline endpoints?
  - `(base_point_xyz, direction_vector_xyz)`?
  - Something else?
Need to cross-reference against the Revit API docs or by inspecting
walls in a known .rvt and comparing decoded values to model inspection.

**Q16** — What does the `variant_marker` byte actually encode? 0x07fa, 0x0821, 0x0870
all appear. Is there a full enumeration? Each variant probably has
different body layout.

**Q17** — Where does wall *type* (thickness, material, layer composition)
come from? It's not in the 32 ArcWall records directly. It must be linked
through HostObjAttr or through a separate WallType/Symbol record.
