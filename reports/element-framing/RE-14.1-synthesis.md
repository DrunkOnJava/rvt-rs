# RE-14.1 synthesis — HostObjAttr (0x006b) signal is bimodal, real records have shared-suffix structure

**Date:** 2026-04-21
**Scope:** Hex-dump 64 B following every `0x006b` occurrence on three partition streams. Compute distance-to-next-same-tag histogram. Column-wise byte histograms at +2..+18 relative to the tag.

## TL;DR

`HostObjAttr` tag 0x006b has a **bimodal signal**:

1. **Text artifact (80%)** — bytes `0x6b 0x00` appearing as part of UTF-16LE
   string content. The 16 bytes following the tag reveal the pattern
   `0x2e 0x00 0x75 0x00 0x6e 0x00 0x69 0x00 0x74 0x00 …` = `".unit…"` in
   UTF-16LE. These are the letter 'k' in parameter-name strings like
   `"…xxxk.units…"`, not record markers.

2. **Real-record minority (~12-20%)** — occurrences where column +2/+3
   are `0x00 0x00` (record prefix, not text continuation) reveal actual
   fixed-field structure. Hex dumps show a recurring suffix
   `76 05 00 00 00 00 ff ff ff ff 94 1d 00 00` across multiple
   occurrences — this is a real field constant + sentinel pattern.

**Net finding:** the RE-11 tag-density signal overweights textual
artifacts. Real HostObjAttr record count is probably **700-1100** on
Einhoven Partitions/0 (20% of 5600 hits), which is much closer to the
ElemTable record count than the raw hit count suggested.

A 25.8% mode at 142 B suggests **fixed-size records** in a large

= 142 B). The text-artifact population has a fixed-stride signal
because parameter strings are of similar length.

The long tail (4 to 110,602 B deltas) is consistent with sparse
record insertions between large text blocks.

## Positive finding: real records show shared structure

Looking at occurrences where column 0 is `0x00` (not `0x2e`) — the
minority ~12% population — hex dumps reveal genuine record structure:

```
Einhoven Partitions/5 occurrences @ 4194, 5403, 5847:
6b 00 00 00 76 05 00 00 00 00 ff ff ff ff 94 1d 00 00 ff ff ff ff ...
```

Same 20-byte suffix across three different occurrences at different
byte offsets. This is definitive evidence of a fixed-field record.

Decoding the suffix:
- `76 05 00 00` = u32 LE 0x00000576 = 1398
- `00 00` = pad?
- `ff ff ff ff` = sentinel (matches the 0xFFFFFFFF marker we've seen elsewhere)
- `94 1d 00 00` = u32 LE 0x00001d94 = 7572
- `00 00` = pad
- `ff ff ff ff ff ff …` = 0xff fill

Additional evidence from Partitions/5 @ 4305 and @ 5514:
```
6b 00 00 00 a8 20 00 00 63 00 00 00 76 05 00 00 00 00 ff ff ff ff 94 1d 00 00 ff ff
6b 00 00 00 b3 20 00 00 63 00 00 00 76 05 00 00 00 00 ff ff ff ff 94 1d 00 00 ff ff
```

These have an extra prefix before the shared suffix. `a8 20` → `b3 20`
(small delta = 11), suggesting this prefix is a variable ID/counter
field, and the record body after is still the fixed suffix.

## Hypotheses

**H9 (confidence 0.8, new)** — HostObjAttr records have a fixed tail
containing two u32 constants (`0x00000576 = 1398` and `0x00001d94 = 7572`)
followed by `0xFFFFFFFF` sentinel(s). These constants are likely
class-identifier values, schema-version stamps, or fixed field IDs.
Before this tail there's a variable prefix containing per-record data
(IDs, references, offsets). Total record size is approximately 30-60
bytes based on observed dumps.

**H10 (confidence 0.9, new)** — The RE-11 signal-over-noise methodology
needs a second-stage filter by the bytes immediately following the tag.
Tags embedded in UTF-16 strings (like `k` before `.unit`) produce
deterministic-looking signal but aren't records. Filtering occurrences
where `buf[pos+2] == 0x00` (record-null) isolates the real-record
population.

**H11 (confidence 0.5, new)** — Because RE-11 found at least 10 tagged
classes with similar strong signals, and because at least one of them
(HostObjAttr) has a bimodal population, likely ALL 10 have this bimodal
structure. Re-ranking by "record-prefix population count" (col 0 =
0x00) rather than "raw count" would give us the true ordering of
element-relevant tags.

## Implications for partition iteration

If HostObjAttr's "real record" count is 20% × 5600 = ~1120 on Einhoven
Partitions/0, and if that correlates 1:1 with architectural elements,
then the Einhoven model has ~1120 architectural elements — which
reconciles much better with the ElemTable record count than the
5600 raw count did.

This changes how `walker::iter_partition_elements()` should work
(task RE-15):

1. For each tagged class suspected of being an element marker, iterate
   only occurrences where col 0 is not a UTF-16-text pattern.
2. For each such "real occurrence", read the fixed field block (the
   0x00000576 / 0x00001d94 constants are structural markers to help
   confirm).
3. Variable prefix carries the per-element data (owner ID, parent ref).

## Decisions

**D13** — Filter HostObjAttr occurrences by `buf[pos+2] == 0x00 && buf[pos+3] == 0x00`
before counting real records. The existing RE-11 probe will be re-run
with this filter in RE-14.2.

**D14** — Extend the probe methodology to all 10 top-signal tags.
Each likely has a similar text-artifact population that must be filtered.

**D15** — Document the shared-suffix constants 0x576 (1398) and 0x1d94
(7572) for later resolution. These are likely schema-referenced
values — they may appear in the `Formats/Latest` decompressed data at
specific offsets as class-ID or field-ID markers. Cross-reference with
the schema parser's output.

## Open questions

**Q13** — What are the specific values 0x576 and 0x1d94? They are NOT
schema class tags (both exceed the 0x01ff range of the 80 tags). They
might be:
  - Field IDs in the element's property bag
  - Type-ID values in a separate namespace
  - Revit-internal category or workset IDs
  Needs cross-reference with the 405-class schema's field declarations.

**Q14** — Is there a variable-length string field in the prefix before
the fixed suffix? The `a8 20 00 00 63 00 00 00` prefix on the fuller
Partitions/5 occurrences decodes as two u32s (0x20a8 = 8360 and 0x63 =
99). These are likely IDs/references.
